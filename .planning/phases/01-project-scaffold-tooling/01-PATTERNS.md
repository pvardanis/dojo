## Scoping Note: Greenfield Phase

The repo currently contains only `.planning/`, `docs/`, `CLAUDE.md`,
and `.git/`. There is no `app/`, `migrations/`, `tests/`, or tooling
config. **Every file in Phase 1 is created from scratch.** Consequently,
there is no in-repo analog for any task — the planner cannot point a
`<read_first>` block at a sibling source file, because no sibling source
exists.

Instead, PATTERNS.md maps each new file to three categories of analog:

1. **Design-spec analog** — `docs/superpowers/specs/2026-04-18-dojo-design.md`
   is authoritative on tree shape (§4.1), Makefile targets (§8.1),
   pre-commit (§8.2), CI (§8.3), and repo-root CLAUDE.md shape (§8.4).
2. **Research drop-in** — `.planning/phases/01-project-scaffold-tooling/
   01-RESEARCH.md` contains **verbatim-paste-ready code** for 16 of the
   files. The planner can put the snippet path + line numbers in the
   task `<action>` block; the executor pastes + verifies.
3. **External convention source** — `~/Documents/Black Lodge/
   knowledge-base/wiki/python-project-setup.md` for ABOUTME headers,
   file-size limits, Protocol-vs-Callable, `get_logger(__name__)`,
   dataclass-for-containers. These are conventions, not code.

**When a file has a RESEARCH.md drop-in, the planner should instruct
the executor: "paste the snippet from 01-RESEARCH.md §<section>,
verify against the surrounding conventions in this PATTERNS.md."**
When no drop-in exists (e.g., `alembic.ini`), the planner must
synthesize from principles and name the canonical external source.

## Pattern Assignments (per-file detail)

### `app/main.py` (composition root, request-response)

**Drop-in location:** 01-RESEARCH.md lines 807-856
**Role:** The ONLY module in the repo allowed to import across layers
(CLAUDE.md composition-root rule).

**Key structural elements the executor must preserve:**

1. **Lifespan hook** configures logging on startup — logging is NOT
   configured at import time (test isolation + fixture control).
2. **`create_app()` factory** — uvicorn targets `app.main:app`; the
   module-level `app = create_app()` binding satisfies this.
3. **Template mount on `app.state`** — keeps the `Jinja2Templates`
   instance out of route-module globals (M9-shaped: no hidden
   module-level side effects at import).
4. **`include_router(home.router)`** — only one router in Phase 1;
   Phase 4+ adds `sources`, `cards`, `drill` via the same pattern.

**Integration points (planner must call out in the task):**
- Imports `app.logging_config.configure_logging` and
  `app.logging_config.get_logger` — this is the ONLY place
  `configure_logging` is called at runtime.
- Imports `app.settings.get_settings` — singleton access pattern.
- Imports `app.web.routes.home.router` — establishes the
  route-include-via-import convention.

**Missing in Phase 1 (will be added in later phases — do NOT add now):**
- DB session dependency (Phase 3+ wires `get_session`).
- Use-case wiring (Phase 4+ wires repositories and use cases).
- Lifespan DB engine disposal (Phase 3+).

### `app/settings.py` (config, load-on-boot)

**Drop-in location:** 01-RESEARCH.md lines 668-740

**Key structural elements:**

1. **`SettingsConfigDict(env_file=".env", extra="ignore")`** — Pydantic
   v2 settings API. `extra="ignore"` means unknown env vars don't crash
   instantiation.
2. **`SecretStr` for `anthropic_api_key`** — `repr()` renders as
   `'**********'`; prevents accidental logging (LLM-03 + New Pitfall 4).
3. **`@lru_cache`'d `get_settings()`** — dep-injection singleton;
   FastAPI's `Depends(get_settings)` gets the same instance; Alembic's
   `env.py` reimports the same cached instance (D-01 single source of
   truth).

**Open question flagged by RESEARCH.md §Assumptions A7 + Open Questions
#1:** should `anthropic_api_key` have a `SecretStr("dev-placeholder")`
default so `make run` works on a fresh clone without `.env`? **Planner
must resolve this at task-writing time** (recommendation in RESEARCH.md
is "yes, add the placeholder default"; alternative is "require real
key"). CI in any case needs a dummy `ANTHROPIC_API_KEY=ci-placeholder`
env var.

### `app/logging_config.py` (infrastructure, cross-cutting)

**Drop-in location:** 01-RESEARCH.md lines 742-805

**Key structural elements:**

1. **`configure_logging(log_level)`** — called ONCE per process
   (structlog's `configure_once` is idempotent; safe if called again).
2. **Env-switched renderer** — `DOJO_ENV=prod` → `JSONRenderer`;
   otherwise `ConsoleRenderer`. Tests get `ConsoleRenderer` but the
   conftest fixture clamps third-party logger levels to WARNING.
3. **`get_logger(name)` helper** — wiki + OPS-04 mandate every module
   does `log = get_logger(__name__)`. This helper is imported, not
   redefined.

**Integration points:**
- `app.main.lifespan` calls `configure_logging(settings.log_level)` once.
- Every future module does `from app.logging_config import get_logger;
  log = get_logger(__name__)`.

### `app/web/routes/home.py` (web route, request-response)

**Drop-in location:** 01-RESEARCH.md lines 864-891

**Key structural elements:**

1. **`APIRouter()` module-level** — ONE router per route-module;
   `app.main` includes it via `include_router`.
2. **Templates accessed via `request.app.state.templates`** — avoids
   the route-module importing `Jinja2Templates` directly (keeps
   Jinja instance a composition-root concern).
3. **`/health` returns a dict** — FastAPI auto-serializes to JSON
   with `JSONResponse`; no manual `json.dumps` needed.
4. **`response_class` declared on each route** — helps FastAPI's
   OpenAPI schema generation and documents intent.

**Template autoescape (FLAG 10 resolution):** Starlette's
`Jinja2Templates(directory=...)` enables `select_autoescape()` for
`.html/.htm/.xml` by default (verified in 01-RESEARCH.md §"Note on
Jinja2 autoescape", line 858). No explicit `autoescape=True` kwarg
needed.

### `app/infrastructure/db/session.py` (infrastructure, DB)

**Drop-in location:** 01-RESEARCH.md lines 348-398

**Key structural elements:**

1. **`class Base(DeclarativeBase)`** — empty in Phase 1. Phase 3 will
   add ORM models that subclass it.
2. **`engine = create_async_engine(settings.database_url, ...)`** —
   reads DB URL from pydantic-settings; does NOT hardcode SQLite (D-01).
3. **`AsyncSessionLocal` with `expire_on_commit=False`** — C3
   mitigation from PITFALLS.md; non-negotiable for async SQLAlchemy.
4. **Dialect-guarded `@event.listens_for(engine.sync_engine, "connect")`**
   — the **only** place SQLite PRAGMAs are set; the `if dialect.name
   != "sqlite": return` guard keeps Postgres (future) unaffected.

**PRAGMAs applied (per D-01 + M12 + FLAG 6):**
- `PRAGMA foreign_keys=ON` — SQLite defaults to OFF; we need ON.
- `PRAGMA journal_mode=WAL` — single-writer concurrency for aiosqlite.
- `PRAGMA busy_timeout=5000` — graceful retry on `SQLITE_BUSY`.

**Integration points:**
- Imported by `migrations/env.py` as `Base` (for `target_metadata`).
- Imported by Phase 3+ repositories (not by any Phase 1 file).
- `get_settings()` import is the ONLY upward-layer import (settings
  is a validation boundary, not a layer).

### `migrations/env.py` (Alembic async env)

**Drop-in location:** 01-RESEARCH.md lines 578-652

**Key structural elements:**

1. **`config.set_main_option("sqlalchemy.url", get_settings().database_url)`**
   — overrides whatever `alembic.ini` contains. D-01 single source of
   truth.
2. **`from app.infrastructure.db.session import Base  # noqa: F401`** —
   the noqa is intentional (M9 defense: importing `Base` will transitively
   import the models module in Phase 3; Phase 1 has no models yet but
   the structural import stays).
3. **`run_async_migrations()`** uses `async_engine_from_config` +
   `async with connectable.connect()` + `await connection.run_sync(
   do_run_migrations)` — the canonical `alembic init -t async` shape.
4. **`asyncio.run(run_async_migrations())` at module top level** — ONLY
   runs when Alembic's CLI invokes env.py as a script. Tests must NOT
   import this module; they use `alembic.command.upgrade` wrapped in
   `asyncio.to_thread` (see conftest; New Pitfall 5).

**Integration points:**
- Imports `app.settings.get_settings` — shared singleton with the
  running app.
- Imports `app.infrastructure.db.session.Base` — shared declarative
  base (Phase 3+ models register against this).
- Consumed by `make migrate` (= `alembic upgrade head`) and by
  `tests/conftest.py::_migrated_engine` (via `command.upgrade`).

### `migrations/versions/0001_initial.py` (empty revision)

**Drop-in location:** 01-RESEARCH.md lines 538-564

**Key structural elements:**

1. **`revision: str = "0001"`, `down_revision: str | None = None`** —
   first revision, no parent.
2. **`upgrade()` body is empty** — Alembic still creates the
   `alembic_version` tracking table on first run. That table is the
   "expected table" satisfying SC #3.
3. **`downgrade()` body is empty** — no-op; satisfies Alembic's
   bidirectional contract.

**Naming:** D-10 allows any stamp; RESEARCH.md uses `0001_initial.py`
via `alembic revision --rev-id=0001 -m "initial"`. Planner can pick a
date-stamp (`20260420_0000-initial`) if preferred; either is acceptable.

### `tests/conftest.py` (pytest fixtures)

**Drop-in location:** 01-RESEARCH.md lines 411-516

**Key structural elements:**

1. **`event_loop_policy` fixture (scope=session)** — pytest-asyncio
   1.x canonical override point (D-05, New Pitfall 1).
2. **`test_db_url` fixture (scope=session)** — uses `tmp_path_factory`

### Pattern: `get_logger(__name__)` per module

**Source:** CLAUDE.md Conventions; `python-project-setup.md`.

**Apply to:** every Python module that does logging (Phase 1:
`app/main.py` only; Phase 2+ adds more).

**Required shape:**

```python
from app.logging_config import get_logger

log = get_logger(__name__)
```

**Anti-pattern:** `import logging; log = logging.getLogger(__name__)` —
DO NOT use stdlib directly. structlog wraps stdlib; the helper gives
us structured context vars.

**Phase 1 status:** only `app/main.py` uses this in Phase 1. The
convention is established here for every future module.

---

### Pattern: Protocol vs Callable (project-local clarifier)

**Source:** CLAUDE.md Conventions + `python-project-setup.md` wiki +
spec §4.1.

**Apply to:** any future port-like abstraction. **Phase 1 has no
ports (D-11 — no `app/application/`).** Pattern is established as a
constraint, not as code to write.

**Rule (verbatim from CLAUDE.md):** if a port is stateless and exposes
one operation, use a typed `Callable` alias (e.g.
`UrlFetcher = Callable[[str], str]`). Reach for `typing.Protocol`
only when the abstraction has state, multiple related methods, or
clear growth pressure.

**Phase 2+ impact:** planner for Phase 2 (`app/domain/`) and Phase 3
(`app/application/ports.py`) must apply this rule. Phase 1 is a no-op
for the pattern itself.

### Pattern: sphinx-style docstrings (100% coverage, interrogate-enforced)

**Source:** CLAUDE.md Conventions + `pyproject.toml` `fail-under = 100`.

**Apply to:** every public module/class/function/method.

**Verification:** `uv run interrogate -c pyproject.toml app`
must report 100.0%.

**D-15 clarifier:** one-line docstrings on Protocol methods are
acceptable. Do NOT add `app/application/ports.py` to interrogate's
`exclude` list. Phase 1 has no Protocols yet, but the pattern is
established now.

---

### Pattern: pristine test output

**Source:** PITFALLS.md m8; `pyproject.toml` `filterwarnings =
["error"]`.

**Apply to:** every test file and fixture.

**Rule:** tests must produce zero warnings and zero unexpected log
lines. Intentional error-triggering tests must capture and assert the
error output (not let it leak to stderr).

**Phase 1 enforcement:**
- `filterwarnings = ["error"]` promotes warnings to test failures.
- `_clamp_third_party_loggers` fixture silences noisy libs at WARNING.
- Verifier note 6 (RESEARCH.md line 1674): temporarily remove the
  clamp and confirm real warnings surface before re-adding. Do this
  once at phase-close time.

---

### Pattern: composition-root is the ONLY cross-layer importer

**Source:** CLAUDE.md Architecture section + spec §4 + wiki
"dependencies flow inward only."

**Apply to:** every non-`app/main.py` module.

**Rule:** domain imports nothing from application/infrastructure/web;
application imports from domain and Protocols only; infrastructure
imports from application (ports) and domain. `app/main.py` is the
only module that wires layers together.

**Phase 1 status:** Phase 1 has no domain/application yet. The rule
applies trivially: `app/main.py` imports from `app/settings`,
`app/logging_config`, `app/web/routes/home`. Nothing imports up the
dependency graph.

**Caveat for `migrations/env.py`:** migrations are tooling, not
application code. env.py legitimately imports from both
`app.settings` and `app.infrastructure.db.session`. RESEARCH.md
§Architectural Responsibility Map line 190 confirms this is a
sanctioned exception.
